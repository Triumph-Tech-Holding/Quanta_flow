fluxos multi-etapa e templates dos 4 tipos de assistente.

- API: REST via React Query v5 + WebSocket via Socket.io
- Deploy: Replit (produção separado do desenvolvimento)
- ORM: Drizzle — todas as mudanças de schema via migration

=== TAREFAS EXATAS A EXECUTAR ===

--- TAREFA 3.1 — IMPLEMENTAR FILA DE JOBS ASSÍNCRONOS ---

IMPORTANTE: NÃO usar Redis ou Bull neste sprint.
O Replit tem limitações de infraestrutura.
Usar solução in-process com Node.js puro.

Implementar um JobQueue simples em memória:

  a) Criar arquivo: server/jobQueue.ts
     - Classe JobQueue com métodos: add(job), process(), start(), stop()
     - Job tem: id, type, payload, runAt (timestamp), status
     - Types de job: 'send_message', 'check_inactivity'
     - Processar jobs a cada 5 segundos via setInterval
     - Jobs com runAt no futuro ficam aguardando
     - Jobs executados são removidos da fila
     - Em caso de erro: marcar job como 'failed', logar erro, continuar

  b) Inicializar o JobQueue no startup do servidor
     - Chamar jobQueue.start() quando o Express iniciar
     - Exportar instância singleton: export const jobQueue = new JobQueue()

  c) Handler do job 'send_message':
     - Recebe: { contactPhone, message, provider, userId }
     - Chama sendMessage() com os parâmetros recebidos
     - Loga sucesso ou falha

  d) Handler do job 'check_inactivity':
     - Recebe: { contactId, conversationId, inactivityTimeout }
     - Verifica se o contato respondeu desde que o job foi criado
     - Se não respondeu: marcar conversa como inativa, logar
     - Se respondeu: cancelar (não fazer nada)

--- TAREFA 3.2 — IMPLEMENTAR DELAY DE RESPOSTA ---

Atualizar processIncomingWhatsAppMessage() para usar o responseDelay
do fluxo encontrado:

  a) Após findMatchingAutomationFlow() retornar um fluxo com match:
     - Ler o campo responseDelay do fluxo (default: 10 segundos)
     - NÃO chamar sendMessage() diretamente
     - Criar job do tipo 'send_message' com runAt = agora + responseDelay segundos
     - Adicionar job ao jobQueue

  b) Após envio da resposta, se inactivityTimeout estiver configurado:
     - Criar job do tipo 'check_inactivity' com
       runAt = agora + inactivityTimeout minutos
     - Adicionar job ao jobQueue

  c) Se chegar nova mensagem do mesmo contato:
     - Cancelar jobs pendentes de 'check_inactivity' desse contato
     - Criar novo job de inatividade com tempo reiniciado

--- TAREFA 3.3 — IMPLEMENTAR FLUXOS MULTI-ETAPA ---

Atualizar o schema e a lógica para suportar sequências de mensagens:

  a) Adicionar campo à tabela automation_flows via migration Drizzle:
     steps   jsonb   (nullable, default: null)

     Estrutura do campo steps:
     [
       { "order": 1, "message": "Olá! Tudo bem?", "delaySeconds": 0 },
       { "order": 2, "message": "Vi que você tem interesse...", "delaySeconds": 30 },
       { "order": 3, "message": "Posso te ajudar com mais detalhes?", "delaySeconds": 60 }
     ]

  b) Atualizar processIncomingWhatsAppMessage():
     - Se o fluxo tem steps preenchidos: usar steps (ignorar responseTemplate)
     - Se o fluxo NÃO tem steps: usar responseTemplate (comportamento atual)
     - Para cada step: criar job 'send_message' com
       runAt = agora + step.delaySeconds segundos

  c) Atualizar o frontend /automation:
     - Adicionar seção "Sequência de Mensagens" no formulário
     - Botão "Adicionar Etapa" cria novo step com campos:
       ordem (automática), mensagem (textarea), delay em segundos (input)
     - Botão "Remover" em cada step
     - Drag-and-drop para reordenar steps (opcional — só se for simples)
     - Se steps estiver vazio: usar campo responseTemplate normalmente
     - Se steps tiver itens: ocultar campo responseTemplate
