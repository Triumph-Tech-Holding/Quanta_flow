Agent, vamos consolidar a infraestrutura completa do Quanta FLOW em uma única implementação. Este prompt inclui: Configurações Persistentes, Anti-Cache Completo e RBAC Enterprise.

OBJETIVO GERAL:
Criar uma infraestrutura profissional, segura e escalável com gerenciamento de credenciais, cache inteligente e controle de acesso baseado em roles.

STACK MANTIDO:
- Backend: Node.js + Express.js
- Frontend: React + Vite
- Banco de Dados: PostgreSQL
- ORM: Drizzle
- Autenticação: JWT com roles
- WebSocket: Socket.io
- Cache: Node.js Memory com TTL

13. Schema (Drizzle) - Tabelas de RBAC:
    
    a) Tabela "roles":
       - id (UUID, primary key)
       - name (VARCHAR, unique) - 'super_admin', 'admin', 'user'
       - description (VARCHAR)
       - created_at (TIMESTAMP)
    
    b) Tabela "permissions":
       - id (UUID, primary key)
       - name (VARCHAR, unique) - ex: 'view_settings', 'edit_settings', 'delete_users'
       - description (VARCHAR)
       - resource (VARCHAR) - ex: 'settings', 'users', 'inbox'
       - action (VARCHAR) - ex: 'view', 'create', 'update', 'delete'
       - created_at (TIMESTAMP)
    
    c) Tabela "role_permissions" (junction):
       - id (UUID, primary key)
       - role_id (UUID, foreign key → roles.id)
       - permission_id (UUID, foreign key → permissions.id)
       - created_at (TIMESTAMP)
    
    d) Tabela "user_roles" (junction):
       - id (UUID, primary key)
       - user_id (UUID, foreign key → users.id)
       - role_id (UUID, foreign key → roles.id)
       - assigned_by (UUID, foreign key → users.id)
       - created_at (TIMESTAMP)
    
    e) Tabela "audit_logs":
       - id (UUID, primary key)
       - user_id (UUID, foreign key → users.id)
       - action (VARCHAR) - ex: 'changed_role', 'accessed_settings', 'deleted_user'
       - resource (VARCHAR) - ex: 'user', 'settings', 'configuration'
       - resource_id (UUID)
       - old_value (TEXT)
       - new_value (TEXT)
       - ip_address (VARCHAR)
       - user_agent (VARCHAR)
       - created_at (TIMESTAMP)
    
    Gere e execute as migrações.

14. Definição de Roles e Permissões (Seed Data):
    
    SUPER_ADMIN (Dono):
    ├─ view_settings, edit_settings, delete_settings
    ├─ view_users, create_users, edit_users, delete_users
    ├─ view_audit_logs, export_audit_logs
    ├─ view_inbox, edit_inbox
    ├─ view_api_configs, edit_api_configs
    └─ manage_roles, assign_roles
    
    ADMIN (Gerente):
    ├─ view_inbox, edit_inbox
    ├─ view_users, create_users, edit_users (não pode deletar super_admin)
    ├─ view_audit_logs (apenas de usuários que gerencia)
    └─ NÃO tem: view_settings, edit_settings, view_api_configs
    
    USER (Atendente):
    ├─ view_inbox, edit_inbox
    ├─ view_leads, create_leads, edit_leads
    └─ NÃO tem: view_users, view_settings, view_audit_logs

15. Middleware de Autenticação e Autorização:
    - Crie arquivo: backend/src/middleware/authMiddleware.js
    - Função: verifyToken(req, res, next)
      • Valida JWT
      • Extrai user_id e roles do token
      • Injeta em req.user
    
    - Crie arquivo: backend/src/middleware/rbacMiddleware.js
    - Função: checkPermission(requiredPermission)
      • Valida se usuário tem permissão
      • Retorna 403 se não autorizado
      • Registra tentativa de acesso não autorizado em audit_logs
    
    - Função: checkRole(requiredRole)
      • Valida se usuário tem role específico
      • Retorna 403 se não autorizado
