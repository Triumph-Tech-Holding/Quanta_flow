SPRINT 5 — QUANTA FLOW
Objetivo: Implementar integrações externas, CI/CD automático
e canais adicionais (Telegram, Instagram, E-mail).

=== CONTEXTO DO SISTEMA ===

Stack:
- Backend: Node.js + Express + Drizzle ORM
- Banco: PostgreSQL (produção) — 49 contatos reais, NÃO pode quebrar
- Frontend: React + Vite + TypeScript + Shadcn UI + Tailwind CSS
- API: REST via React Query v5 + WebSocket via Socket.io
- Deploy: Replit (produção separado do desenvolvimento)
- ORM: Drizzle — todas as mudanças de schema via migration

=== O QUE FOI ENTREGUE NOS SPRINTS ANTERIORES ===

SPRINT 1 (concluído):
- findMatchingAutomationFlow() conectada ao webhook
- detectIntent() chamada automaticamente a cada mensagem recebida
- sendMessage() disparada automaticamente quando há match de keyword

SPRINT 2 (concluído):
- Schema atualizado com 9 novos campos em automation_flows
- Frontend /automation com formulário das 7 seções completas

SPRINT 3 (concluído):
- JobQueue in-memory em server/jobQueue.ts
- Delay de resposta e inatividade via jobQueue
- Fluxos multi-etapa com campo steps (jsonb)
- 4 templates: Qualificador, Resolutivo, Transacional, Especialista

SPRINT 4 (concluído):
- Fila de leads com queueStatus (waiting/assigned/resolved)

- SLA com slaDeadline, slaBreached e job 'check_sla'

=== TAREFAS EXATAS A EXECUTAR ===

--- TAREFA 5.1 — WEBHOOKS PERSONALIZADOS DE SAÍDA ---

Implementar sistema de webhooks de saída para notificar
sistemas externos sobre eventos do Quanta Flow:

  a) Criar nova tabela outbound_webhooks via migration Drizzle:
     id              serial primary key
     userId          integer     (FK → users.id)
     name            text        (nome amigável do webhook)
     url             text        (URL de destino)
     events          jsonb       (lista de eventos: ex:
                                 ['lead.created', 'lead.qualified',
                                  'conversation.closed',
                                  'flow.success', 'flow.interrupt'])
     isActive        boolean     (default: true)
     secret          text        (nullable, para validação HMAC)
     createdAt       timestamp   (default: now())

  b) Criar endpoints no backend:
     GET    /api/webhooks/outbound         → lista webhooks do usuário
     POST   /api/webhooks/outbound         → cria novo webhook
     PUT    /api/webhooks/outbound/:id     → atualiza webhook
     DELETE /api/webhooks/outbound/:id     → remove webhook
     POST   /api/webhooks/outbound/:id/test → envia payload de teste

  c) Criar server/webhookDispatcher.ts:
     - Função dispatchEvent(event, payload, userId)
     - Busca todos os outbound_webhooks ativos do userId
       que escutam o evento recebido
     - Para cada webhook: faz POST HTTP com o payload
     - Headers: Content-Type: application/json
     - Se secret configurado: adicionar header
       X-Quanta-Signature com HMAC-SHA256 do payload
     - Timeout: 5 segundos por chamada
     - Em caso de falha: logar erro, não travar o fluxo principal
     - NÃO usar retry automático neste sprint

  d) Injetar dispatchEvent() nos pontos de evento:
     - Após lead criado: dispatchEvent('lead.created', contact, userId)
     - Após stage mudar para 'qualificado':
       dispatchEvent('lead.qualified', contact, userId)
     - Após successCondition atingida:
       dispatchEvent('flow.success', { contact, flow }, userId)
     - Após interruptCondition atingida:
       dispatchEvent('flow.interrupt', { contact, flow }, userId)
     - Após conversa encerrada:
       dispatchEvent('conversation.closed', conversation, userId)

  e) Criar página /settings/webhooks no frontend:
     - Lista de webhooks configurados com status ativo/inativo
     - Formulário: nome, URL, eventos (checkboxes), secret (opcional)
     - Botão "Testar webhook" que chama POST /api/webhooks/outbound/:id/test
     - Exibir último status de envio (sucesso/falha) por webhook

--- TAREFA 5.2 — INTEGRAÇÃO COM GOOGLE SHEETS ---

  a) Criar nova tabela sheet_integrations via migration Drizzle:
     id              serial primary key
     userId          integer     (FK → users.id)
     name            text        (nome amigável)
     spreadsheetId   text        (ID da planilha Google)
     sheetName       text        (nome da aba, default: 'Leads')
     triggerEvent    text        (evento que dispara: 'lead.created'
                                 | 'lead.qualified' | 'flow.success')
     columnMapping   jsonb       (mapeamento campo → coluna:
                                 ex: { "name": "A", "phone": "B",
                                       "stage": "C", "score": "D" })
     isActive        boolean     (default: true)
     googleToken     text        (nullable, token OAuth2 criptografado)

  b) Criar server/googleSheetsService.ts:
     - Usar biblioteca googleapis (já disponível no Node.js)
     - Função appendRow(spreadsheetId, sheetName, rowData, token)
       que adiciona linha na planilha
     - Função getAuthUrl(userId) que gera URL de autorização OAuth2
     - Função handleOAuthCallback(code, userId) que troca code
       por token e salva criptografado em googleToken

  c) Criar endpoints no backend:
     GET  /api/integrations/sheets              → lista integrações
     POST /api/integrations/sheets              → cria integração
     PUT  /api/integrations/sheets/:id          → atualiza integração
     DELETE /api/integrations/sheets/:id        → remove integração
     GET  /api/integrations/sheets/auth         → retorna URL OAuth2
     GET  /api/integrations/sheets/callback     → handler OAuth2 callback

  d) Integrar com webhookDispatcher:
     - Quando dispatchEvent() for chamado, verificar se há
       sheet_integration ativa para o evento
     - Se sim: chamar appendRow() com os dados mapeados em columnMapping
     - Executar de forma assíncrona (não bloquear o fluxo principal)

  e) Criar página /settings/integrations no frontend:
     - Card "Google Sheets" com botão "Conectar conta Google"
     - Após autenticação: formulário para configurar a integração
     - Campos: nome, ID da planilha, nome da aba, evento gatilho,
       mapeamento de colunas (interface visual simples)
     - Status de conexão (conectado/desconectado)
     - Botão "Desconectar" que revoga o token

--- TAREFA 5.3 — HANDLER TELEGRAM ---

Implementar recebimento e envio de mensagens via Telegram,
reaproveitando toda a lógica já construída para WhatsApp:

  a) Criar endpoint no backend:
     POST /api/webhooks/telegram
     - Receber updates do Telegram Bot API
     - Extrair: chat.id, from.id, from.first_name, text
     - Normalizar para o mesmo formato interno do WhatsApp:
       { phone: chat.id, name: from.first_name, message: text,
         channel: 'telegram' }
     - Chamar processIncomingMessage() com os dados normalizados
       (refatorar processIncomingWhatsAppMessage() para aceitar
        qualquer canal — ver item 5.3c)

  b) Criar server/telegramService.ts:
     - Função sendTelegramMessage(chatId, message, botToken)
       que faz POST para https://api.telegram.org/bot{token}/sendMessage
     - Função registerWebhook(botToken, webhookUrl)
       que registra a URL do webhook no Telegram

  c) Refatorar processIncomingWhatsAppMessage():
     IMPORTANTE: NÃO reescrever a função — apenas generalizar:
     - Renomear para processIncomingMessage(data, channel)
     - Parâmetro channel: 'whatsapp' | 'telegram' | 'instagram' | 'email'
     - Toda a lógica interna permanece igual
     - Apenas o sendMessage() final deve usar o canal correto:
       se channel = 'whatsapp' → Z-API/Baileys
       se channel = 'telegram' → telegramService.sendTelegramMessage()

  d) Adicionar campo channel à tabela conversations via migration:
     channel   text   (nullable, default: 'whatsapp')

  e) Criar página /settings/channels no frontend:
     - Card "Telegram" com campo para inserir Bot Token
     - Botão "Conectar" que chama registerWebhook()
     - Status de conexão (conectado/desconectado)
     - Instruções de como criar um bot no BotFather
