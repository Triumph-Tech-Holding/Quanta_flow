Agent, vamos implementar um sistema de RBAC (Role-Based Access Control) profissional no Quanta FLOW para gerenciar diferentes níveis de acesso com segurança enterprise.

OBJETIVO:
Criar uma infraestrutura de controle de acesso baseado em roles onde cada usuário tem permissões específicas, com auditoria completa e proteção de endpoints críticos.

STACK MANTIDO:
- Backend: Node.js + Express.js
- Frontend: React + Vite
- Banco de Dados: PostgreSQL
- ORM: Drizzle
- Autenticação: JWT com roles inclusos

TAREFAS DE BACKEND:

1. Schema (Drizzle) - Tabelas de RBAC:
   
   a) Tabela "roles":
      - id (UUID, primary key)
      - name (VARCHAR, unique) - 'super_admin', 'admin', 'user'
      - description (VARCHAR)
      - created_at (TIMESTAMP)
   
   b) Tabela "permissions":
      - id (UUID, primary key)
      - name (VARCHAR, unique) - ex: 'view_settings', 'edit_settings', 'delete_users'
      - description (VARCHAR)
      - resource (VARCHAR) - ex: 'settings', 'users', 'inbox'
      - action (VARCHAR) - ex: 'view', 'create', 'update', 'delete'
      - created_at (TIMESTAMP)
   
   c) Tabela "role_permissions" (junction):
      - id (UUID, primary key)
      - role_id (UUID, foreign key → roles.id)
      - permission_id (UUID, foreign key → permissions.id)
      - created_at (TIMESTAMP)
   
   d) Tabela "user_roles" (junction):
      - id (UUID, primary key)
      - user_id (UUID, foreign key → users.id)
      - role_id (UUID, foreign key → roles.id)
      - assigned_by (UUID, foreign key → users.id)
      - created_at (TIMESTAMP)
   
   e) Tabela "audit_logs":
      - id (UUID, primary key)
      - user_id (UUID, foreign key → users.id)
      - action (VARCHAR) - ex: 'changed_role', 'accessed_settings', 'deleted_user'
      - resource (VARCHAR) - ex: 'user', 'settings', 'configuration'
      - resource_id (UUID)
      - old_value (TEXT)
      - new_value (TEXT)
      - ip_address (VARCHAR)
      - user_agent (VARCHAR)
      - created_at (TIMESTAMP)
   
   Gere e execute as migrações.

2. Definição de Roles e Permissões (Seed Data):
   
   SUPER_ADMIN (Dono):
   ├─ view_settings, edit_settings, delete_settings
   ├─ view_users, create_users, edit_users, delete_users
   ├─ view_audit_logs, export_audit_logs
   ├─ view_inbox, edit_inbox
   ├─ view_api_configs, edit_api_configs
   └─ manage_roles, assign_roles
   
   ADMIN (Gerente):
   ├─ view_inbox, edit_inbox
   ├─ view_users, create_users, edit_users (não pode deletar super_admin)
   ├─ view_audit_logs (apenas de usuários que gerencia)
   └─ NÃO tem: view_settings, edit_settings, view_api_configs
   
   USER (Atendente):
   ├─ view_inbox, edit_inbox
   ├─ view_leads, create_leads, edit_leads
   └─ NÃO tem: view_users, view_settings, view_audit_logs

3. Middleware de Autenticação e Autorização:
   - Crie arquivo: backend/src/middleware/authMiddleware.js
   - Função: verifyToken(req, res, next)
     • Valida JWT
     • Extrai user_id e roles do token
     • Injeta em req.user
   
   - Crie arquivo: backend/src/middleware/rbacMiddleware.js
   - Função: checkPermission(requiredPermission)
     • Valida se usuário tem permissão
     • Retorna 403 se não autorizado
     • Registra tentativa de acesso não autorizado em audit_logs
   
   - Função: checkRole(requiredRole)
     • Valida se usuário tem role específico
     • Retorna 403 se não autorizado

4. Endpoints de RBAC (Admin):
   - GET /api/admin/roles - listar roles
   - GET /api/admin/permissions - listar permissões
   - POST /api/admin/users/:userId/roles - atribuir role a usuário
   - DELETE /api/admin/users/:userId/roles/:roleId - remover role
   - GET /api/admin/audit-logs - listar logs de auditoria
   - Todos requerem: checkPermission('manage_roles')

5. Proteção de Endpoints Críticos:
   - /api/admin/settings/* → checkPermission('edit_settings')
   - /api/admin/users/* → checkPermission('manage_users')
   - /api/admin/audit-logs/* → checkPermission('view_audit_logs')
   - /api/inbox/* → checkPermission('view_inbox')
   - /api/leads/* → checkPermission('view_leads')

6. JWT com Roles:
   - Ao fazer login, incluir no JWT:
     {
       "userId": "uuid",
       "email": "user@example.com",
       "roles": ["user"],
       "permissions": ["view_inbox", "edit_inbox", "view_leads"]
     }
   - Expiração: 24h
   - Refresh token: 7 dias

7. Seed Script (Usuário Mestre):
   - Crie arquivo: backend/scripts/seedAdmin.js
   - Criar usuário:
     • Email: admin@quantashop.com.br
     • Senha: Quanta@2026 (hashed com bcrypt)
     • Role: super_admin
   - Executar com: npm run seed:admin
   - Adicionar ao package.json: "seed:admin": "node scripts/seedAdmin.js"

8. Auditoria Completa:
   - Registrar em audit_logs:
     • Mudança de role de usuário
     • Acesso a endpoints sensíveis
     • Falhas de autenticação
     • Mudanças em configurações
     • Deletions de usuários
   - Incluir: user_id, action, resource, old_value, new_value, ip_address, timestamp
