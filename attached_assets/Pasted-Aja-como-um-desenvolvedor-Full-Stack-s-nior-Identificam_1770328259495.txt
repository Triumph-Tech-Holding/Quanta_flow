Aja como um desenvolvedor Full Stack sênior. Identificamos que o dashboard não está atualizando devido ao cache HTTP do navegador, que está recebendo status 304 Not Modified e ignorando os dados novos do servidor.

OBJETIVO:
Implementar um sistema inteligente de cache que desabilita cache para dados dinâmicos (APIs) mas mantém cache para assets estáticos (CSS, JS, imagens).

PROBLEMA IDENTIFICADO:
- Dashboard recebe status 304 Not Modified
- Navegador ignora dados novos do servidor
- Usuários veem dados desatualizados
- Problema ocorre em todas as requisições de API

STACK MANTIDO:
- Backend: Node.js + Express.js
- Frontend: React + Vite
- Banco de Dados: PostgreSQL
- WebSocket: Socket.io

TAREFAS DE BACKEND:

1. Middleware Anti-Cache para APIs (CRÍTICO):
   - Crie arquivo: backend/src/middleware/antiCacheMiddleware.js
   - Função: antiCacheMiddleware(req, res, next)
   - Headers a adicionar em TODAS as respostas de API:
     
     res.set({
       'Cache-Control': 'no-store, no-cache, must-revalidate, proxy-revalidate, max-age=0',
       'Pragma': 'no-cache',
       'Expires': '0',
       'Surrogate-Control': 'no-store',
       'ETag': 'W/"' + Date.now() + '"' // Força novo ETag a cada requisição
     });
     
     next();

2. Middleware Seletivo (IMPORTANTE):
   - Não aplique anti-cache a TUDO
   - Crie dois middlewares:
     
     a) antiCacheMiddleware: para /api/* (dados dinâmicos)
     b) cacheMiddleware: para /public/* (assets estáticos)
   
   - Posicionamento no app.js:
     
     // 1. Middleware de CORS (antes de tudo)
     app.use(cors({
       origin: process.env.FRONTEND_URL,
       credentials: true,
       methods: ['GET', 'POST', 'PUT', 'DELETE'],
       allowedHeaders: ['Content-Type', 'Authorization']
     }));
     
     // 2. Middleware Anti-Cache para APIs
     app.use('/api', antiCacheMiddleware);
     
     // 3. Middleware de Cache para Assets
     app.use('/public', express.static('public', {
       maxAge: '1d',
       etag: false
     }));
     
     // 4. Rotas
     app.use('/api', routes);

3. Remover ETag de Respostas Dinâmicas:
   - Adicione ao antiCacheMiddleware:
     
     res.removeHeader('ETag');
     res.set('ETag', 'W/"' + Date.now() + '"');
   
   - Isso força o navegador a sempre fazer nova requisição

4. Validação de Headers CORS:
   - Certifique-se que CORS não está conflitando:
     
     ✅ CORRETO:
     cors({
       origin: 'https://seu-app.vercel.app',
       credentials: true,
       methods: ['GET', 'POST', 'PUT', 'DELETE'],
       allowedHeaders: ['Content-Type', 'Authorization']
     })
     
     ❌ ERRADO:
     cors() // sem configuração = aceita tudo
   
   - Validar que Access-Control-Allow-Origin está correto

5. Versionamento de API (RECOMENDADO):
   - Adicione versão nas rotas:
     
     app.use('/api/v1', antiCacheMiddleware);
     app.use('/api/v1/inbox', inboxRoutes);
     app.use('/api/v1/users', userRoutes);
   
   - Isso permite manter versões antigas se necessário

6. Socket.io - Desabilitar Cache:
   - No arquivo de configuração do Socket.io:
     
     const io = require('socket.io')(server, {
       cors: {
         origin: process.env.FRONTEND_URL,
         credentials: true
       },
       transports: ['websocket', 'polling'],
       pingInterval: 25000,
       pingTimeout: 60000
     });
   
   - WebSocket não usa cache HTTP (já é real-time)
   - Mas configure transports para evitar fallback a polling com cache

7. Logs de Debug:
   - Adicione logs no antiCacheMiddleware:
     
     console.log(`[${new Date().toISOString()}] ${req.method} ${req.path}`);
     console.log(`Cache-Control: ${res.get('Cache-Control')}`);
   
   - Isso ajuda a validar que headers estão sendo enviados

8. Teste de Headers (CRÍTICO):
   - Crie endpoint de teste: GET /api/test-cache
   - Retorne JSON com timestamp:
     
     res.json({
       timestamp: new Date().toISOString(),
       message: 'Se o timestamp muda a cada requisição, cache está desabilitado'
     })
   
   - Teste no navegador: abra DevTools → Network → veja status 200 (não 304)

TAREFAS DE FRONTEND:

1. Remover Cache do Fetch/Axios:
   - Adicione header em TODAS as requisições:
     
     axios.defaults.headers.common['Cache-Control'] = 'no-cache';
     axios.defaults.headers.common['Pragma'] = 'no-cache';
   
   - Ou configure no interceptor:
     
     api.interceptors.request.use(config => {
       config.headers['Cache-Control'] = 'no-cache';
       config.headers['Pragma'] = 'no-cache';
       return config;
     });

2. Adicionar Timestamp em Requisições:
   - Força nova requisição mesmo com cache:
     
     const response = await axios.get('/api/inbox', {
       params: { t: Date.now() } // timestamp como query param
     });
   
   - Exemplo: /api/inbox?t=1704067200000

3. Validar Status da Resposta:
   - No frontend, verifique se recebe 200 (não 304):
     
     api.interceptors.response.use(response => {
       console.log(`Status: ${response.status}`); // deve ser 200
       return response;
     });

4. Limpar Cache do Navegador (Desenvolvimento):
   - Adicione ao App.jsx:
     
     useEffect(() => {
       // Limpar cache ao montar componente
       if (process.env.NODE_ENV === 'development') {
         caches.keys().then(names => {
           names.forEach(name => caches.delete(name));
         });
       }
     }, []);

REQUISITOS TÉCNICOS CRÍTICOS:

1. Ordem de Middlewares (OBRIGATÓRIO):
   - CORS deve vir ANTES de anti-cache
   - Anti-cache deve vir ANTES das rotas
   - Ordem correta:
     1. CORS
     2. Body Parser
     3. Anti-Cache
     4. Rotas
     5. Error Handler

2. Diferenciação de Rotas:
   - /api/* → anti-cache (dados dinâmicos)
   - /public/* → cache 1 dia (assets estáticos)
   - /webhooks/* → anti-cache (webhooks)
   - /static/* → cache 30 dias (imagens, CSS, JS)

3. Validação de Headers:
   - Abra DevTools → Network → clique em requisição
   - Verifique Response Headers:
     ✅ Cache-Control: no-store, no-cache, must-revalidate
     ✅ Pragma: no-cache
     ✅ Expires: 0
     ✅ ETag: W/"timestamp"

4. Status HTTP Esperado:
   - ✅ Primeira requisição: 200 OK
   - ✅ Segunda requisição: 200 OK (não 304)
   - ❌ Se receber 304: cache ainda está ativo

5. Ambiente Replit:
   - Replit pode ter cache em nível de servidor
   - Após fazer mudanças, clique "Republish"
   - Aguarde 30 segundos para propagação
   - Teste em aba anônima (sem cache local)
